Start here

What you are about to operate

The Stage Workflow Runner is a manifest-driven runner for high-stakes, staged OpenAI Responses workflows. It treats every artifact as evidence — schema-versioned, hashed, and confined to a single workspace root — so that a tar of that root is a complete, replayable audit trail.

Read this sentence three times

High-stakes — the design assumes an answer could be wrong in a way that costs money or trust, so every claim must trace to a hashed file. Staged — work is split into ordered stages with review gates between them. Manifest-driven — you define a workflow with JSON + Markdown in a task pack, never with Python. The engine is generic; the task pack is the workflow.

How to use this page

This is built to take you from first principles to a complete mental model, step by step. Everything is interactive — diagrams light up, steppers walk you through sequences, cards expand. Go in order the first time:

01 · First principles

Six ideas everything derives from

Click each card. If you understand only one section of this whole explainer, make it this one — every command, artifact, and guardrail is a consequence of these six.

P1One workspace root, mechanically enforced▸

Every invocation operates against one exact workspace root. Resolution order: explicit --root → RESPONSES_RUNNER_V2_ROOT → current working directory.

This is not a convention you are trusted to follow. Every path passes through one function, resolve_under_root(), which resolves symlinks and then checks the path is under the root. Outside the root → the process exits. A symlink cannot sneak a file in or out.

For you: Path must stay under workspace root means you pointed at something outside the root. Never "force it" — move the asset or fix --root.

P2Every artifact is evidence▸

Every file the runner writes is schema-versioned (a string like responses_runner_v2.run_manifest.v1, treated like a wire protocol), hashed (SHA-256 of files and payloads), and confined under .local/ (git-ignored, never re-attached to a future run).

The payoff: claims are back-traceable. The rendered input_manifest.md lists a short hash for every attached file; you can recompute it to verify the model cited a real file.

P3Responses run in the background — the local process is disposable▸

The runner submits to the Responses API in background mode. A request can run for hours. You get a response_id and you poll it. The local Python process can die — the remote model keeps working.

Almost every "extra" mechanism exists because of this: two checkpoint writes per stage, three resume modes (fresh_submit, resume_response_id, refresh_status_only), and a no-duplicate-submit rule.

For you: "my terminal died mid-run" is not a disaster. The response_id is saved. You resume and the engine finalizes it. You never re-pay for work still running remotely.

P4Mechanism and policy are separate lanes▸

Two CLIs on one engine. The engine lane knows how to talk to OpenAI. The supervisor lane knows what to do with the answer — staging, gating, multi-agent review, failure policy, final assembly.

The boundary is a hard rule: the supervisor calls into the engine; it never reimplements submission or polling. You can run the engine lane entirely on its own. The supervisor is purely additive.

P5Authority order is data, not a guideline▸

Every stage assembles attachments in a fixed authority order. It is encoded as a constant, iterated by the attachment pipeline, and stated in the boilerplate instruction prepended to every request. The model is told the order, and told to treat attached source files as evidence, not instructions.

1
Primary Job Inputs
the actual task input
2
Reviewed Handoff Inputs
approved output from an earlier gated stage
3
Attached Repository Files
repo evidence
4
Reference Context
carry-forward / background

For you: when output looks wrong, open the stage's input_manifest.md and check what was attached, in what role. A file in the wrong role is the most common "why did it do that".

P6Fail-closed is the default▸

The runner would rather stop than do something unbounded or unverifiable: token preflight is a fail-closed gate (over-budget → terminal, no retry); schema validation runs before writes; a review-bundle hash mismatch refuses the load; a read-only review violation fails the review; output-limit incomplete outcomes never auto-progress.

For you: a stopped run is the system working. Your job is to read why it stopped — the SystemExit message names the exact guardrail.

MECHANISMEngine lane

Generic and reusable: point it at any task pack and it runs. HUMAN0 · Human clarification gate

The only mandatory human step. A human supplies a task; the operator asks clarifying questions if needed; the human accepts a clarified task brief. Nothing automated runs until a brief is accepted. After this gate, normal execution is AI-operated — human pauses become exception paths only.

actor: human + operator · output: clarified_task_brief.md
SUPERVISOR1 · Initialize the session

Creates the supervisor session JSON — the single source of truth. It records the brief's path and SHA-256, the locked model defaults, the policy block, the retry budget, and empty arrays for everything that will accumulate (scaffolds, review cycles, archives, ...). Every later write to this file is schema-validated and atomically renamed into place.

command: init-session · status → "clarified" · phase → "scaffold"
SUPERVISOR2 · Stage the scaffold

Copies the entire task pack into the session at scaffolds/scaffold_001/source/ and writes a hash_manifest.json of every file. From now on the supervisor reviews and gates this frozen, hashed copy — not your working tree.

command: stage-scaffold · status → "scaffold_staged"
GATE3 · Examine the scaffold (STATIC gate)

The static pre-launch gate. It validates without constructing a Stage 1 request: the workflow manifest schema, that the model posture matches the locked defaults, that there is exactly one terminal stage and it is last, that every non-terminal stage is review-gated, that sidecar / output schemas use no unsupported keywords (if / then / oneOf / allOf / not / ...), the tool profiles, the attachment inventory, and that each stage prompt is substantive. Blocking issues → session "blocked".

command: examine-scaffold · emits scaffold_examination.{json,md}
GATE4 · Dry-run the scaffold (EXECUTABLE gate)

The executable gate. It calls the engine's run_workflow(dry_run=True): the engine resolves inputs and writes a request payload, but never calls the API. Where examine-scaffold proves the scaffold is well-formed, dry-run-scaffold proves a real Stage 1 request can actually be constructed from it.

command: dry-run-scaffold · calls the engine with dry_run=True
SUPERVISOR5 · Scaffold review cycle

The full 3-agent review loop, applied to the scaffold itself. The accountable operator Codex runs first and writes a provisional decision; the independent read-only Codex and Claude reviewers run next; consolidation merges findings (advisory only); the operator records a selective-acceptance decision with applied-change evidence. See section 06 for the full loop.

review_kind: scaffold · operator → reviewers → consolidate → accept
GATE6 · Launch allowed?

The launch check passes only when the latest scaffold version is marked "accepted" AND an accepted scaffold review cycle exists. This is the boundary: everything before it is preparation; everything after it spends tokens. By the time Stage 1's request reaches OpenAI, the workflow has been validated four different ways.

check: assert_scaffold_launch_allowed
ENGINE7 · Run Stage N (LIVE)

The first thing that ever reaches OpenAI. The engine resolves attachments, renders + uploads the input manifest, writes the request payload, runs token preflight, POSTs to /responses, writes checkpoint #1, polls to a terminal status, writes response.final.*, extracts structured output and runs the sidecar if configured, then writes checkpoint #2. Walk the full loop in section 05.

command: run --wait · the engine's 12-step stage loop
SUPERVISOR8 · Classify the outcome

The supervisor inspects the stage checkpoint and response markdown and assigns exactly one of six classifications, each with its own (reviewable, bundle-able, rerun) triple and recovery action. See section 07. Only completed_complete_artifact and failed_complete_artifact are reviewable.

command: classify · emits a stage_outcome.v1 record
SUPERVISOR9 · Stage review cycle → approved bundle

If the stage is reviewable, run the same 3-agent loop on its output (review_kind = stage_output). On an approving acceptance record, create-bundle produces the approved review bundle — the contract the next stage consumes as Reviewed Handoff Inputs. Then loop back to node 7 for the next stage.

review_kind: stage_output · ends with create-bundle
SUPERVISOR10 · Finalize the implementation bundle

The terminal artifact. finalize-bundle schema-validates the packet. The schema requires that file_inventory[] and emitted_files[] are the identical set of paths — the "no half-finished implementations, no hidden files" guarantee — plus all three agent reviews, an approving operator_acceptance record, validation evidence, a model-migration summary, a failure-policy summary, a human-pause summary, rollout instructions, and residual risks.

command: finalize-bundle · session status → "completed"
If you only run the engine lane

The whole left side collapses to: put the task pack under the root, then run --dry-run. That dry-run is your single pre-flight check. The supervisor lane is what adds staging, the two gates, and automated multi-agent review on top.

Scaffold review cycle
supervisor · review_kind = scaffold

The full 3-agent loop on the scaffold itself: invoke-operator → invoke-reviewers → consolidate → accept. The operator must supply applied-change evidence for anything accepted, or the supervisor auto-rejects it.

gate · the boundary between prep and spend

Passes only when the latest scaffold version is "accepted" AND an accepted scaffold review cycle exists. Crossing this line is the first time the system will spend tokens. Everything before it was free.

Two of these steps are gates you can fail

Examine-scaffold (static) and dry-run-scaffold (executable) can both block the session. And step 5 is a full review loop. By the time Stage 1's request reaches OpenAI, the workflow has been validated four different ways.

1 · Resolve attachments
engine · attachments.py

Merge the static input manifest with operator-supplied inputs, review-bundle inputs, and carry-forward artifacts. Walk directories (skipping .git, .local, node_modules, ...), apply exclude globs, hash every file, and check the 50 MB / 100-file limits — all before any HTTP traffic.

2 · Render + write input_manifest.{json,md}
engine · the model's table of contents

The rendered input_manifest.md enumerates every attached file, its authority role, and a short SHA-256. It is the human-readable source of truth for "what was attached" — and your first stop when output looks wrong.

3 · Write request_payload.json
engine · --dry-run RETURNS HERE

The exact body that will be sent to /responses is assembled and written to disk. With --dry-run, the engine stops here: nothing is uploaded, no API is called. This is your cheapest validation.

4 · Upload every file
engine · multipart /files

Every file — including input_manifest.md, uploaded first — is pushed via the multipart files endpoint. Non-context file types that are UTF-8 text get wrapped as Markdown first. The file-id ↔ source-path mapping is recorded in uploads.json.

5 · Token preflight
engine · fail-closed gate (skip with --skip-token-count)

The engine can call POST /responses/input_tokens. If the count exceeds a configured max_input_tokens, the stage is marked blocked and the process exits — terminal, no retry, no fallback. Unbounded token spend is treated as a safety property.

6 · POST /responses → checkpoint #1
engine · background mode

The request is submitted. In background mode it comes back almost immediately as queued with a response_id. The engine writes response.latest.json and the first stage checkpoint — already enough to resume from if the local process dies.

7 · Poll until terminal (if --wait)
engine · rewrites response.latest.json each poll

With --wait, the engine polls the response_id until it reaches completed / failed / cancelled / incomplete, rewriting response.latest.json on every poll. Without --wait, it returns now and you resume later.

8 · Write response.final.{json,md}
engine · artifacts.py

On a terminal status the engine writes the finalized raw response and a human-readable Markdown render — token usage, cached/reasoning tokens, source citations, tool-call summaries, and the upload lifecycle.

9 · Extract structured output / run sidecar
engine · sidecar.py (only on the terminal-artifact path)

If the stage requested json_schema output, structured output is extracted. If a sidecar is configured, a second Responses call runs: it re-reads the primary markdown + JSON and asks the structural model for a strict-schema payload → output.structured.json + sidecar.response.*.

10 · Write checkpoint #2 + update run_manifest
engine · the durable resume point

The second, final stage checkpoint records the terminal status and pointers to every artifact. The run manifest's per-stage summary is updated with statuses and artifact hashes.

11 · Map response status → stage status
engine · workflow state machine

A completed response on a review_required stage with a next stage becomes stage status waiting_for_review — not completed. The run will not proceed past it until you supply --review-bundle.

12 · Auto-progress?
engine · needs ALL FIVE conditions

The engine loops to the next stage only when: the stage completed, there IS a next stage, the gate is auto, you passed --wait, and you did NOT pin a specific --stage. Anything else stops the run after one stage.

How the request is assembled

instructions = COMMON_RUNNER_INSTRUCTIONS   # the authority-order boilerplate
             + shared_instructions.md       # task-pack-wide
             + stage_instructions.md        # optional, per stage

input = [ stage task prompt text ]
      + [ "role: Stage Input Manifest"      → input_manifest.md ]  # model TOC, uploaded first
      + [ "role: Primary Job Inputs"        → files... ]
      + [ "role: Reviewed Handoff Inputs"   → files... ]
      + [ "role: Attached Repository Files" → files... ]
      + [ "role: Reference Context"          → files... ]   # strict authority order
Attaching any file type

A .go / .rs / .tsx / .sql file is not a Responses-API context type. So if it is UTF-8 text, the engine wraps it as Markdown — source path in front matter, body in a fenced block. That is how the runner can attach any text file. Limits enforced during resolution (before any HTTP traffic): 50 MB per file, 50 MB per request, 100 files per request.

06 · The review loop

Three independent agents, one operator decision

For every scaffold and every non-terminal stage, three agents review the same materials. Step through the loop.

1 · invoke-operator
operator_codex · codex exec · accountable, may apply changes

The accountable operator Codex runs first and writes a provisional review decision. For scaffold / stage_output / final_packet / recovery cycles, this provisional record is required before the independent reviewers may run.

2 · invoke-reviewers
codex_review_agent + claude_review_agent · read-only, snapshot-enforced

The Codex reviewer (codex exec) and the Claude reviewer (claude -p --model opus --effort max) run independently and read-only. The supervisor snapshots every workspace source file before and after each one. Any change to a file other than the agent's own output sidecars is a read_only_violation that fails the review.

3 · consolidate
consolidation_pass · deterministic · ADVISORY ONLY

A deterministic merge of the operator + two reviewer findings, de-duplicated by (recommendation text + affected artifacts). It emits the same review_decision.v1 shape with next_action: proceed_to_operator_acceptance. It accepts nothing — it only organizes.

4 · accept
operator · selective acceptance with applied-change evidence

The operator selects which recommendation IDs to accept and supplies an applied-change-evidence JSON. The supervisor rewrites any "accepted" recommendation to "rejected" unless its evidence has concrete changes_applied[], validation_evidence[], and operator_rationale. A rejected blocking recommendation flips the approval to do_not_approve.

5 · create-bundle
supervisor · only on an approving acceptance record

An approved review bundle is created only if the acceptance record's approval_decision == "approve". The bundle references the approved markdown, the raw response JSON, and reviewer notes — each with its SHA-256 — and becomes the next stage's Reviewed Handoff Input.

The operator cannot rubber-stamp

An accepted recommendation without concrete changes_applied[] + validation_evidence[] + operator_rationale is auto-rejected by the supervisor itself. The supervisor does not synthesize evidence. A rejected blocking recommendation becomes a blocking issue, which flips the approval to do_not_approve.

One protocol for all of them

Every agent — operator, both reviewers, and the consolidation pass — emits the same JSON shape: review_decision.v1. actor_role distinguishes them. That uniformity is what makes the outputs comparable. Read-only enforcement is a before/after hash snapshot of the whole workspace — a reviewer that edits a file is caught automatically. The Claude lane deliberately does not use --bare (which would skip OAuth); the supervisor strips API-key env vars so the local subscription login is used, and falls back from --effort max to --effort xhigh once if needed.

07 · When a stage fails

Six outcomes, six recovery paths

When a stage reaches a terminal state, the supervisor's classify command assigns exactly one of six classifications. Click each to see its recovery path. The flags are reviewable / bundle-able / rerun.

HAPPY PATHcompleted_complete_artifact

The response completed and produced complete, substantive markdown + JSON artifacts (and the sidecar, if configured). Action: review — proceed through the normal operator / Codex / Claude / consolidation / acceptance loop. Reviewable ✓ · bundle-able ✓.

REVIEWABLEfailed_complete_artifact

The response reported failed, but it still produced a substantive artifact (markdown ≥ 40 chars, not the empty placeholder). Action: review_failed_artifact — treat it as reviewable, preserve the failed status, review and bundle only after acceptance. Reviewable ✓ · bundle-able ✓.

RECOVERfailed_no_artifact

The response failed and produced no substantive artifact. Action: archive_before_rerun. No review, no bundle. You must archive-attempt first; the rerun is allowed only if the request and scaffold hashes are unchanged and retry budget remains (default: 1).

HUMAN PAUSEincomplete_output_limit

The response ran out of output tokens. Action: block_and_recover + a human pause. It never auto-progresses. A human must decide on scope, model, or budget before any rerun.

HUMAN PAUSEblocked_token_preflight

Token preflight failed closed, or the checkpoint is blocked. Action: human_pause. No live submission, no bundle. Repair the scaffold or input set, re-run dry-run validation, and relaunch only after approval.

HUMAN PAUSElong_running_monitoring_anomaly

A live response stayed queued / in_progress past the stale threshold. Action: monitor_without_duplicate_submit + a human pause. Refresh or resume the existing response_id — never submit duplicate work while it may still complete.

The dividing line between the two failed_* outcomes

It is "substantive markdown": the response markdown must be ≥ 40 characters and not literally "No assistant text was returned." A failed response with substantive text is still reviewable; without it, you must archive before any rerun.

Archive-before-rerun — the anti-"quietly changed the inputs" guard

A failed_no_artifact stage may be rerun only after archiving. The archive records a deterministic request_hash and scaffold_hash. Before the rerun is allowed, the supervisor checks the current hashes against the archive's _before hashes. If you changed the prompt and tried to claim a "fresh failure," the hashes won't match — the rerun is blocked.

08 · The workflows

Workflow modes & the bundled packs

A workflow manifest declares a workflow_mode; the loader enforces the stage count. Three gate types control progression between stages.

The four workflow modes

ModeStagesTypical gate pattern
one_passexactly 1terminal
two_passexactly 2auto → terminal
reviewed_three_stageexactly 3review_required → review_required → terminal

The three gate types

  • auto — the engine may auto-progress to the next stage (only when you passed --wait, didn't pin a --stage, and there is a next stage).
  • review_required — the engine stops. The output ends in status waiting_for_review and must be reviewed and packaged into an approved review bundle before the next stage can run.
  • terminal — the last stage. There is exactly one, and it is last.

Carry-forward: how a stage sees the previous stage

  • reference_context_from_stage_ids — attaches the prior stage's response.final.md under the lowest-authority Reference Context role. Cheap, no review required.
  • review_bundle_from_stage_id — requires you to supply that stage's approved review bundle with --review-bundle. Its contents land under Reviewed Handoff Inputs. This is the gated path.

Carry-forward is deliberately cheap — it attaches the actual approved text, not a lossy summary.

The locked model posture

The loader rejects anything else at load time: primary generation gpt-5.5-pro, structural processing gpt-5.5, GPT-5.5-family prompt cache retention 24h, high-stakes reasoning effort xhigh (structural high/medium), locked max output 128000.

The diagnosis habit

A SystemExit from any CLI is designed to name the exact guardrail that tripped. Read it literally before changing anything. A stopped run is the system protecting the evidence chain.

10 · Glossary

The sixteen words that run everything

Click any term to expand it. The rest of the system is just these, combined.

01Task pack▸

A directory of JSON + Markdown that defines a workflow: workflow manifest, per-stage input manifests, prompts, tool profiles, output schemas, shared instructions. No Python.

02Workflow manifest▸

The *.workflow.json file. Declares the workflow id, mode, model roles, request defaults, the ordered stage list, each stage's gate, carry-forward rules, and optional sidecar extraction.

03Stage▸

One unit of work = one Responses API call (plus an optional sidecar call). Stages are ordered by stage_number.

04Gate▸

A stage's progression rule: auto (engine may continue), review_required (stop — output must be reviewed and bundled first), terminal (last stage).

05Input manifest▸

The per-stage *.input_manifest.json. Declares the static attachment set, grouped into the four authority roles. The engine merges it with operator inputs and carry-forward at runtime, then renders input_manifest.md.

06Authority order▸

The fixed ranking of attachment roles: Primary Job Inputs → Reviewed Handoff Inputs → Attached Repository Files → Reference Context. Encoded as data, stated in every request.

07Review bundle▸

The approval contract passed between gated stages. A JSON file referencing the approved markdown, raw response JSON, and reviewer notes — each with its SHA-256. Loading it re-hashes everything and refuses on mismatch.

08Run manifest▸

run_manifest.json — the top-level durable state of one run: workflow id, the workflow manifest's hash, the ordered stage list, and a per-stage summary.

09Stage checkpoint▸

stage_checkpoint.json — the durable, resumable state of one stage: status, the response_id, token-preflight result, and pointers to every artifact. Written twice per stage.

10Sidecar▸

An optional second Responses call after a stage completes. It re-reads the stage's markdown and extracts a strict-JSON-schema payload using the structural model — keeping creative generation and machine-ingestible structure as separate passes.

11Supervisor session▸

The supervisor's single source of truth: a schema-validated supervisor_session.json plus a directory of review cycles, scaffolds, archives, monitoring records, and the final bundle.

12Scaffold▸

A task pack staged into a supervisor session and hashed, so the supervisor can gate it — examine, dry-run, review — before any live stage runs.

13Review cycle▸

One pass of the supervisor's review loop on a scaffold, a stage output, a final packet, or a recovery path: operator → 2 independent reviewers → consolidation → operator acceptance.

14Resume mode▸

How a checkpoint was produced: fresh_submit (first run), resume_response_id (rehydrate a stored id and finalize), refresh_status_only (just poll, do not finalize).

15Token preflight▸

A fail-closed gate: before live submission the engine can count input tokens; over the configured limit is terminal — no retry, no fallback. Skipped with --skip-token-count.

16Final implementation bundle▸

The supervisor's terminal artifact. Schema-validated; requires file_inventory and emitted_files to be the identical set of paths, all three agent reviews, and an approving operator acceptance.
